# Onboarding — human-training

Welcome. You're in the **workflow infrastructure** project — the repo that
*produces* the `human-training` Claude Code plugin. The skills shipped by
this plugin (project-genesis, lifecycle-manager, handover-manager, etc.) are
meant for **downstream hobby projects** — they are not the workflow you
follow when working *in this repo*.

**Read this entire doc before doing anything.** It is short. The first
section in particular will save you from a class of mistakes that looks
sensible but isn't.

---

## ⚠️ Meta-confusion — read this first

This repo's relationship to its own skills is inverted from the normal case.

- **The skills in `skills/` are products.** They are built artifacts shipped
  to other projects via the plugin system. Treat `skills/` as build output,
  not as source.
- **The sprint-based workflow described inside those skills (DEVLOG, spec,
  Sprint 1..N, confidence bar) is for downstream projects.** This repo
  doesn't have a DEVLOG, doesn't run sprints, doesn't have a `spec.md`.
  Don't invent one.
- **Don't invoke `human-training:*` skills to do work *in* this repo** —
  with one exception: the user can ask you to use `onboarding-creator` on
  this repo (that's how this very file was authored, dogfood-style). For
  everything else, work directly.
- **Don't apply the office-tour pattern recursively.** This *is* the office
  tour. There is no deeper room.

What this repo actually is: a small build pipeline that turns two kinds of
source material (`workflow/guides/` and `skills-source/`) into a shared
`skills/` package, then exposes that package through Claude Code
(`.claude-plugin/plugin.json`) and Codex (`.codex-plugin/plugin.json`)
plugin manifests.

## Getting oriented

The authoritative reference for repo architecture lives in
[README.md](README.md). Read it before this section if you haven't. The
quick orientation:

```
workflow/guides/*.md      Track 1 source — model-agnostic skill bodies
                           (frontmatter synthesized by build scripts)

skills-source/<name>/      Track 2 source — fully-formed Claude-specific
                           skills, copied through to skills/ as-is

skills-drafts/<idea>/      In-progress skill development. Tracked in git
                           but NOT built and NOT shipped. Scratchpad.

skills/<name>/             BUILD OUTPUT. Never hand-edit. Regenerated by
                           scripts/build-skills.{ps1,sh}. CI verifies it
                           matches a fresh build.

scripts/build-skills.ps1   Windows builder
scripts/build-skills.sh    Linux/macOS builder (also what CI runs)
scripts/verify-plugin-manifests.py
                           Validates Claude/Codex manifests and version parity
.claude-plugin/plugin.json Claude Code plugin manifest
.codex-plugin/plugin.json  Codex plugin manifest; points at the same skills/
.github/workflows/         verify-skills.yml — `diff -r` of committed
                           skills/ against a fresh build plus manifest checks
```

User-facing communication style lives in the user's global CLAUDE.md (auto-
loaded in Claude Code). Defer to that — don't restate it here.

## The workflows that actually happen here

These are the workflows future sessions will be summoned for. They are NOT
sprints. They are discrete operations on the plugin pipeline.

### 1. Wiring a session-authored skill (skills-source/ → ship)

**Trigger:** "I have a new skill in skills-source/, wire it in." Or:
"another session packaged up a skill, here it is."

The user develops a skill in some other session (e.g., a project where the
context is rich for that skill's domain), packages it as `<name>/SKILL.md`
+ any assets/evals, and drops the folder into `skills-source/<name>/`.

Your job:
1. Validate `SKILL.md` frontmatter: `name:` matches the dir name, `description:`
   is present and tuned for trigger accuracy, `allowed-tools:` if specified is
   well-formed.
2. Run `./scripts/build-skills.ps1` (Windows) — confirm the new skill appears
   under `skills/<name>/`.
3. Verify the build-verify invariant: a fresh sh build's output must be
   byte-identical to the committed `skills/`. Run the diff:
   ```bash
   OUTPUT_DIR=/tmp/skills-rebuild bash scripts/build-skills.sh
   diff -r skills /tmp/skills-rebuild
   ```
   Anything but empty output means CI will fail.
4. Bump `version` in both plugin manifests (semver: feature add = minor).
5. Update `README.md` skills table if it makes sense (cosmetic, not required).
6. Show the user the staged changes. Don't `git commit`/`git push` unless
   they explicitly ask.

### 2. Developing a skill *in this repo* (skills-drafts/ → skills-source/)

**Trigger:** "Let's build the foo skill here." Or: "I've been hitting the
same footgun on the X API every project — let's turn it into a skill."

There's no session-with-perfect-context elsewhere; the skill needs to be
built here from raw material.

Your job:
1. Confirm or create `skills-drafts/<idea-name>/`. If raw material already
   exists, look at what's there.
2. Iterate on `skills-drafts/<idea-name>/SKILL.md` until the user is satisfied.
   While iterating, that folder can hold research, notes, candidate triggers,
   sketches of evals — anything. Nothing here ships.
3. **Graduate when ready:**
   - Move (or selectively copy) `skills-drafts/<idea-name>/` →
     `skills-source/<idea-name>/`. Drop research/notes you don't want shipping
     in the plugin — only what the skill needs at runtime travels.
   - Hand off to workflow #1 above.

See [skills-drafts/README.md](skills-drafts/README.md) for the convention
inside that folder.

### 3. Editing a Track 1 (workflow-derived) skill

**Trigger:** "Update the lifecycle skill to mention X." Or: "fix a bug in
the genesis skill."

These five skills are derived from `workflow/guides/*.md`:
`lifecycle-manager`, `handover-manager`, `onboarding-creator`,
`project-genesis`, `workflow-orientation`.

Your job:
1. Edit the source: `workflow/guides/<source>.md` (the markdown body) and/or
   the `$skillDefinitions` entry in **both** `scripts/build-skills.ps1` and
   `scripts/build-skills.sh` (frontmatter: name, description, allowed-tools,
   assets). The two builders must stay in lockstep — CI compares their output.
2. Run the build, verify both builders produce identical output (see
   workflow #1 step 3).
3. Bump `version` in both plugin manifests.
4. Stage changes, show the user.

### 4. Build script / pipeline changes

**Trigger:** "Add support for binary assets," "the build is wrong about X,"
"add a new source track."

Your job:
1. Edit both `scripts/build-skills.ps1` AND `scripts/build-skills.sh`. They
   must produce byte-identical output. This is the CI invariant — break it
   and every push fails.
2. Adjust `.github/workflows/verify-skills.yml` if the set of watched paths
   needs to change.
3. Test the round-trip:
   ```powershell
   ./scripts/build-skills.ps1
   ```
   ```bash
   rm -rf /tmp/skills-rebuild
   OUTPUT_DIR=/tmp/skills-rebuild bash scripts/build-skills.sh
   diff -r skills /tmp/skills-rebuild   # must be empty
   ```
4. Update [README.md](README.md) if the architecture changed visibly.
5. Bump version if the build change affects plugin contents.

### 5. Plugin version bump + multi-machine update

After any change that affects the plugin's `skills/` contents, the version
in both `.claude-plugin/plugin.json` and `.codex-plugin/plugin.json` MUST
bump to the same value. The Claude bump is the signal Claude Code uses to
detect an update. The Codex bump keeps local/package metadata aligned and
lets Codex reinstall/cache-refresh flows detect a changed plugin.

Bump convention: minor for additions, patch for tweaks, major for breaking.

After commit + push, the user runs `/plugin update human-training@human-training`
per Claude Code machine. Codex installs may need a plugin reinstall/cache
refresh and a new thread.

## The invariant: builders match, commit matches build

There is exactly one structural rule for this repo:

> **A fresh `bash scripts/build-skills.sh` into a temp dir must produce
> a tree byte-identical to the committed `skills/`, and both plugin
> manifests must be valid and version-aligned.**

Everything else flows from that. The PowerShell builder must produce the
same bytes as the sh builder. Both must produce the same bytes regardless
of the source's line endings (they normalize to LF / no BOM). CI's
`verify-skills.yml` enforces this on every push and PR, and also runs
`scripts/verify-plugin-manifests.py`.

If you change anything in the pipeline, this is the test you run before
declaring done. Treat it like tests passing in a normal project.

## Things that look reasonable but aren't

- **Editing `skills/` directly.** It's build output. Will be overwritten,
  and CI will fail before you push.
- **Adding a skill by writing only a SKILL.md and dropping it into `skills/`.**
  Same problem. Use `skills-source/` for ready-to-ship session-authored
  skills, or `skills-drafts/` for in-progress.
- **Trying to derive a `workflow/guides/<name>.md` for every new skill.**
  Track 1 is for skills that *genuinely* have a model-agnostic workflow-doc
  form. Claude-specific skills (subagents, Claude Code tooling, etc.)
  belong in Track 2 — the SKILL.md *is* the artifact.
- **Editing only one of the two build scripts.** They must stay in lockstep
  byte-for-byte. If you change the PS one, change the sh one too, and re-
  verify diff parity.
- **Skipping the version bump.** Without it, other machines never see the
  change.
- **Running `git commit`/`git push` without being asked.** This is a shared
  remote. Stage changes, show the user, let them push.

## When the user wants the plugin updated (any machine)

This repo distributes itself: it is **both** the plugin
(`.claude-plugin/plugin.json`) and its own marketplace
(`.claude-plugin/marketplace.json`, `source: "./"`). GitHub is the single
source of truth — no machine-specific paths, no hand-copying. But a push is
**not** instantly live, because Claude Code reads through two local caches per
machine, not the repo directly:

1. **Source of truth** — the GitHub repo. Bumping `version` in both manifests
   is the signal that an update exists.
2. **Marketplace catalog** — a local clone Claude Code keeps. The "is there an
   update?" check compares against *this*, not GitHub. `autoUpdate` is on, so it
   refreshes at Claude Code **startup** — but a long-running session or a failed
   pull leaves it stale. That is exactly how a machine sits many versions behind
   while reporting "you're on the latest."
3. **Installed plugin** — the actual loaded skills. Refreshing the catalog does
   **not** install the new version; that is a separate step.

**To update a machine** (machine-independent — no absolute paths):

```bash
claude plugin marketplace update human-training      # force-refresh the catalog
claude plugin update human-training@human-training   # install the new version
# then FULLY quit and relaunch Claude Code — a new thread/session is not enough
```

The marketplace-refresh line is the one people forget. Without it, `plugin
update` checks a stale catalog and says you're already current — the #1 source
of "I bumped the version but Claude doesn't see it." `autoUpdate` is meant to do
that refresh automatically at startup, but force it explicitly whenever a push
isn't showing up.

**Shortcut:** [`update-plugin.bat`](update-plugin.bat) at the repo root runs both
lines in one step — run it on any machine, then relaunch.

**Fast local iteration (no commit/push):** launch `claude --plugin-dir .` from
the repo root to load the plugin straight from the working copy.

**User-scope shadowing:** a skill installed at user scope
(`~/.claude/skills/<name>/`) shadows the plugin's copy of the same name. If a
skill was developed standalone before being added here, delete the user-scope
copy once the plugin ships it — otherwise the prefix-less version shows up
alongside the plugin's `human-training:`-prefixed one.

## Writing a handover

If you're handing off mid-task to a fresh session in this repo, write a
handover (`HANDOVER.md` at the repo root). Cover:

1. **Which of the five workflows above** you were in (wiring / developing /
   editing-track-1 / pipeline-change / version-bump).
2. **Where you are in that workflow** — what's done, what's next.
3. **Any pending invariant check** — has `diff -r skills /tmp/skills-rebuild`
   been run since your last edit? Has `scripts/verify-plugin-manifests.py`
   passed? Has the version been bumped in both manifests?
4. **What's in working state vs in-flight** — uncommitted edits, half-moved
   folders in `skills-drafts/` or `skills-source/`, etc.
5. **Anything the user decided but isn't in the code yet** — naming choices,
   architectural calls made in-conversation.

Keep it under one screen. The repo is small and most context is in the
files themselves; the handover only needs the ephemeral conversation delta.

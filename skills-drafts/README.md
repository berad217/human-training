# skills-drafts/

In-progress skill development workspace. Nothing here ships in the plugin —
this is where skills get *built*, not where they get bundled.

## ⚠️ Two kinds of folder live here — they look identical, they aren't

A folder in `skills-drafts/<name>/` is one of two things:

1. **A genuine draft** — a skill still being built, *not yet shipped*.
2. **Post-graduation residue** — non-shipping scratch (research, eval
   harnesses, test files) left behind when a skill graduated. The `SKILL.md`
   moved to `skills-source/`; only the scratch stayed.

**The one rule that tells them apart:** *is there a matching
`skills-source/<name>/`?*

- **Yes** → this folder is just **residue**. The skill already ships. Leave it
  (the scratch is kept on purpose — re-runnable evals, source notes) or delete
  it if it's no longer useful. It is **not** unfinished work.
- **No** → this is a **genuine draft**. It does not ship anywhere yet.

This ambiguity is the price of the graduation ritual below (move `SKILL.md`
out, leave research behind). If you're staring at this folder wondering "why is
half-finished stuff still here?" — run the check above before assuming anything
is broken or abandoned.

*Current residue (skills that shipped): `antigravity-cli-workspace/`,
`gemini-api/research/`, `pdf-toc-splitter/`. Current genuine draft: `ollama/`.*

## What goes here

- Raw research and reference material (model cards, API docs, transcripts)
- Half-formed SKILL.md drafts being iterated on
- Notes, links, scratch, lists of footguns to capture
- Test queries / eval data being collected before the skill is finalized

## What does NOT go here

- Production-ready skills → `skills-source/<name>/` (Track 2)
- Workflow-derived skills → `workflow/guides/<name>.md` (Track 1)
- Build output → `skills/` (regenerated by build scripts)

## Convention

One subdir per skill-idea: `skills-drafts/<idea-name>/`. Inside, organize
however helps. A loose pattern:

```
skills-drafts/
└── gemini-api/                       # one folder per skill-idea
    ├── README.md                     # notes: scope, footguns to capture, plan
    ├── research/                     # raw material — model cards, API docs
    │   ├── gemini-3.1-flash-lite.md
    │   └── whats-new-gemini-3.5.md
    ├── eval-queries.md               # candidate triggers (for the eventual trigger-eval.json)
    └── SKILL.md                      # the draft, iterated in place
```

## Build-time guarantees

- Both `build-skills.ps1` and `build-skills.sh` ignore this directory entirely.
- CI (`verify-skills.yml`) does not watch it.
- A draft cannot accidentally leak into the shipped plugin.

## Graduation ritual

When a draft is ready to ship:

1. Polish `SKILL.md` — valid frontmatter, `name:` matches the dir name, the
   `description:` is tuned for trigger accuracy.
2. Move the `<name>/` folder from here to `skills-source/<name>/`. Drop the
   research/notes if you don't want them shipping in the plugin — only what
   the skill actually needs at runtime should travel.
3. Run `./scripts/build-skills.ps1`.
4. Bump `version` in both `.claude-plugin/plugin.json` and `.codex-plugin/plugin.json`.
5. Commit, push, `/plugin update human-training@human-training`.
